# Daily Report Orchestrator

## Global Rules

- You coordinate the end-to-end content curation workflow.
- You must ensure agents are invoked in the correct sequence.
- All final reports must be saved in the `/reports/` directory with the filename `Daily_Report_YYYY-MM-DD.html`.
- **Formatting**: Only HTML should be produced; Markdown versions are deprecated.
- **Versioning**: NEVER overwrite an existing report.
- No content should be generated by the orchestrator; all content comes from agents.
- **Transparency**: When invoking any agent, instruct it to wrap its reasoning in a `<thinking>` block before producing its final output.

## Workflow Sequence

1. **Curator** (`agents/curator/SKILL.md`): Search and gather raw content for the 3 domains. Output must conform to `CuratorOutput` in `schemas/agent_contracts.json`.
2. **Summarizer** (`agents/summarizer/SKILL.md`): Distill the raw content into high-density summaries. Output must conform to `SummarizerOutput`.
3. **Filter** (`agents/filter/SKILL.md`): Remove duplicates and irrelevant noise. Rank remaining items. Output must conform to `FilterOutput`.
4. **Formatter** (`agents/formatter/SKILL.md`): Apply the `dashboard_template.html` and premium styling. Output must conform to `FormatterOutput`.
5. **URL Validation**: Run `scripts/validate_urls.ps1` against the generated HTML. Log results. If validation fails for an article, remove it from the report.
6. **Arbiter** (`agents/arbiter/SKILL.md`): Evaluate the final HTML report. Output must conform to `ArbiterOutput`.
7. **RALPH Loop** *(conditional)*: If `ArbiterOutput.pass` is `false`, execute the targeted correction loop (see below).
8. **Manifest Update**: Run `scripts/build_manifest.ps1` to rebuild `reports_manifest.json`.
9. **Structure Test**: Run `scripts/test_report_structure.ps1` as the final gate.

## RALPH Loop — Targeted Self-Correction

This loop runs **once** at Step 7 if the Arbiter returns `pass: false`. It is limited to **1 retry** to control token cost.

### Process
1. **Read**: Ingest `ArbiterOutput.findings[]` and identify each finding's `responsible_agent`.
2. **Analyze**: Group findings by `responsible_agent` and `severity`. Skip `low` severity findings.
3. **Log**: Append a retry entry to `evals/arbiter/retry_log.md` recording the date, failing agent(s), and finding descriptions.
4. **Plan**: Determine the minimum re-invocation chain needed to resolve the failures:
   - `formatter` failures (bad HTML, section order, stock table) → Re-run **Formatter** only.
   - `curator` failures (bad URLs, missing data) → Re-run **Curator → Summarizer → Filter → Formatter**.
   - Multiple responsible agents → Use the upstream-most agent to anchor the re-run chain.
5. **Help**: Pass the Arbiter's specific findings to the re-invoked agent as an added `<correction>` block in the prompt.
6. After the re-run, invoke the **Arbiter** once more. Do NOT loop again regardless of the second score.

### Correction Prompt Block (injected into the re-invoked agent)
```
<correction>
The previous run of this agent produced the following failures identified by the Arbiter:
{findings[].description for all findings with responsible_agent = this agent}
Please address these specific issues in your output.
</correction>
```
